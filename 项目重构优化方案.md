# 课程排课系统重构优化方案

> 文档目的：为当前毕业设计项目提供一份可执行、可跟踪、可分阶段落地的系统重构方案。
>
> 适用范围：当前根目录下旧版项目（Spring Boot + Vue 2 的前后端分离实现）。
>
> 暂不纳入本轮改造：`newCourseArrange/` 目录内代码**先不优化、不改动**，仅作为后续重构参考。

---

## 1. 当前项目现状概览

通过对项目目录、核心配置、代表性前后端代码、数据库脚本的初步审阅，当前系统存在以下典型问题：

### 1.1 前端现状

- 旧版技术栈偏老：`Vue 2.5 + webpack 3 + Element UI 2.x`，工程配置陈旧，可维护性较弱。
- 缺少工程化质量保障：没有 `ESLint`、`Prettier`、单元测试、E2E 测试、提交规范。
- 接口调用方式分散：大量组件内直接写死 `http://localhost:8080/...`。
- 页面职责过重：多个 `.vue` 文件体量过大，列表、表单、弹窗、接口请求、业务逻辑全部耦合在单文件内。
- 路由设计不规范：子路由大量使用绝对路径；路由守卫中存在重复调用 `next()` 的风险。
- 认证与状态管理薄弱：直接从 `localStorage` 读写用户信息和 token，缺少统一鉴权、过期处理、权限建模。
- 缺少可复用基础设施：未形成统一请求封装、错误提示、分页表格、表单校验、常量字典、hooks/工具函数。
- UI 设计缺乏统一规范：样式分散、配色与间距未形成设计 token，组件命名和目录结构不统一。

### 1.2 后端现状

- 技术栈较旧：`Spring Boot 2.2.4.RELEASE`、`Java 8`、`Swagger 2`、`MyBatis-Plus 3.2.0`。
- 依赖存在升级与安全风险：如 `fastjson 1.2.78`、较老版本 Druid、Swagger2 等。
- 分层不清晰：Controller 中存在较多业务逻辑、参数拼装、查询拼接、返回结构处理。
- 领域模型污染：Entity 直接作为接口入参与出参使用，存在“数据库模型 = 接口模型”的问题。
- 返回结构较原始：统一响应 `ServerResponse` 虽然存在，但未形成标准错误码、异常体系、分页结构、字段语义规范。
- 鉴权实现不完整：JWT 只做了解码，未进行可靠验签与过期校验；权限粒度不足。
- 安全问题明显：密码疑似明文存储、明文比较、明文修改。
- 代码风格不统一：存在 `System.out.println`、魔法值、原始 `QueryWrapper`、命名不规范、注释格式不统一等问题。
- 部分类职责过重：例如排课相关服务/工具类超长，算法、数据装配、数据库访问耦合严重。
- 缺少完整测试：当前旧版后端基本无测试代码。

### 1.3 数据库现状

- 表结构命名风格不统一：同时存在 `tb_xxx` 与无前缀表，命名策略混乱。
- 字段设计语义不足：如 `remark`、`type`、`power`、`piority` 等字段表达不清晰。
- 约束不足：从现有脚本看，明显缺少外键、唯一约束、业务唯一索引、检查约束设计。
- 安全字段设计不规范：密码字段长度与语义不适配现代加密存储。
- 冗余与派生字段较多：存在可推导字段直接落库、状态字段含义不统一的问题。
- 审计字段不统一：`create_time`、`update_time`、`modified_time` 并存。
- 字典与枚举设计不足：课程属性、账号状态、用户类型等大量依赖硬编码。

### 1.4 工程管理现状

- 根目录下同时存在旧项目、文档、日志、目标产物、重构中的 `newCourseArrange`，边界不清晰。
- 缺少统一开发文档：启动方式、环境变量、接口契约、数据库初始化、部署流程未系统沉淀。
- 缺少 CI/CD、代码扫描、自动测试、质量门禁。
- 产物目录如 `target/`、日志目录进入工作区，影响仓库整洁度。

## 2. 重构总目标

本次重构不是“局部修补”，而是以**规范化、可维护、可扩展、可交付**为目标，对项目进行系统性治理。

### 2.1 前端目标（遵循 Vue 规范与前端开发规范）

- 建立清晰的目录结构、组件分层、路由分层、状态管理规范。
- 形成统一的 API 请求层、错误处理机制、权限控制机制。
- 统一 UI 风格、表单校验、分页查询、弹窗交互、消息反馈。
- 降低页面复杂度，将超大组件拆解为页面组件、业务组件、基础组件。
- 建立可持续开发能力：Lint、格式化、测试、构建、环境配置、Mock、文档。

### 2.2 后端目标（遵循阿里巴巴 Java 开发规范）

- 重建清晰分层：`controller / application / domain / infrastructure` 或至少规范化 `controller / service / mapper / dto / vo`。
- 统一接口规范、异常处理、日志规范、参数校验、权限校验。
- 消除明文密码、弱鉴权、魔法值、超大类、重复代码、N+1 查询。
- 将排课算法与通用 CRUD、文件上传、用户中心等业务模块解耦。
- 提升可测试性、可扩展性、可维护性。

### 2.3 数据库目标（遵循数据库设计规范）

- 统一命名规范、主键规范、外键/索引/唯一约束规范。
- 消除歧义字段、冗余字段和硬编码枚举。
- 建立基础字典体系与审计字段规范。
- 为排课场景建立更清晰的核心实体模型。

## 3. 本轮重构边界与原则

### 3.1 范围定义

本轮方案优先覆盖以下内容：

- 旧版前端：`UI/coursearrange/`
- 旧版后端：`src/main/java/`、`src/main/resources/`
- 数据库设计：`doc/数据库备份/db_course_arrangement.sql`
- 项目文档与工程治理：根目录、`doc/`

本轮暂不直接改造：

- `newCourseArrange/` 内现有代码
- 无关的历史产物整理之外的大规模删除操作

### 3.2 重构原则

1. **先建规范，再做迁移**：先统一结构、命名、边界、编码规范，再进入业务改造。
2. **先止血，再优化**：优先修复高风险问题，如安全、鉴权、接口设计、核心数据模型。
3. **先基础设施，再功能迁移**：先搭建公共能力，再迁移具体页面与接口。
4. **保持阶段可运行**：每个阶段结束后都要保证系统最小可运行。
5. **文档同步更新**：每完成一类改造，必须同步更新本方案、设计文档与接口文档。

### 3.3 完成标记约定

后续执行重构时，已完成事项统一使用以下格式标记：

- `[x] ~~已完成事项~~`
- `[ ] 待完成事项`

## 3.4 截至 2026-03-19 的实际进度校准

基于当前仓库中的前后端代码、测试代码与构建配置，现将原方案中的完成状态校准如下：

- 前端已完成 `Vue 3 + Vite + Element Plus + Pinia + Vue Router 4` 升级，且已形成 `api / views / layouts / router / stores / constants / styles / utils` 的主目录结构。
- 前端已具备统一请求层、统一 token 注入、统一错误拦截、登录失效跳转、基于权限上下文的菜单渲染与路由拦截。
- 前端已完成大量旧管理页从硬编码 `http://localhost:8080` 到统一 API 模块的迁移，但仍存在页面拆分、测试、交互一致性和少量旧接口残留问题。
- 后端已升级到 `Spring Boot 2.7.18 + Java 17 + MyBatis-Plus 3.5.7`，并切换为 `jar` 打包。
- 后端已接入 `Sa-Token`、全局异常处理、统一错误码、分页查询模型、部分标准化资源接口，以及一批针对排课任务/课表链路的单元测试。
- `tb_class_task`、`tb_course_plan` 两条旧表运行链已经从主流程中剥离，标准排课任务与标准课表查询/调课链路已能独立运行。
- 排课算法本体仍未完成最终重构。当前虽然已引入 `SchedulingTaskInput` 作为标准排课输入 DTO，但算法内部仍保留大量历史基因编码与 `ClassTask` 过渡模型语义。
- 安全改造仍未彻底完成。密码已引入 `BCrypt`，但旧 JWT 与兼容登录链路尚未完全删除，且部分旧账号仍保留“明文平滑迁移”逻辑。
- 数据库、文档、测试、工程治理仍有明显缺口，当前状态属于“基础设施与主链已基本站稳，正在收尾旧兼容并准备进入算法重构”。

## 4. 建议的目标架构

### 4.1 前端目标架构

建议未来将旧版前端重构为现代化 Vue 工程，推荐目标如下：

- 框架：`Vue 3`
- 构建工具：`Vite`
- 语言：优先 `TypeScript`，至少关键模块使用类型化接口定义
- 路由：`Vue Router 4`
- 状态管理：`Pinia`
- UI 组件：可延续 `Element Plus` 或选择更适合后台系统的现代组件库
- 网络请求：统一 `axios` 封装
- 质量工具：`ESLint + Prettier + Stylelint + Husky + lint-staged`
- 测试：`Vitest` + `Vue Test Utils`，必要时补充 `Playwright`

> 注：是否一步到位迁移到 Vue 3，可在正式实施前再最终确认；但从长期维护看，建议直接升级而不是继续加固旧版 Vue 2 工程。

### 4.2 后端目标架构

建议后端重构为清晰分层的 Spring Boot 应用：

- 框架：`Spring Boot 3.x`（若受限于依赖可先定为 2.7.x，最终再升级）
- JDK：建议 `Java 17`
- 持久层：`MyBatis-Plus` 或 `MyBatis`，但要明确边界，不混乱使用 AR 模式
- 接口文档：`springdoc-openapi`
- 参数校验：`jakarta.validation` / `hibernate-validator`
- 安全：`Spring Security + JWT` 或至少标准化 JWT 鉴权过滤器
- 日志：`Slf4j + Logback`
- 测试：`JUnit 5 + Mockito + SpringBootTest`
- 数据库变更管理：`Flyway` 或 `Liquibase`

### 4.3 数据库目标分层

建议将数据库围绕“学期、年级、班级、课程、教师、教室、排课任务、排课结果、用户权限”重新抽象：

- 基础主数据：学期、年级、班级、课程、教师、教学楼、教室
- 业务计划数据：开课任务、班级授课计划、教师任课关系、固定时间约束
- 排课结果数据：课表结果、冲突记录、排课批次、排课日志
- 系统管理数据：用户、角色、权限、操作日志、文件资源

## 8. 验收标准

重构完成后，建议至少满足以下标准：

### 8.1 前端验收标准

- 页面目录清晰、组件职责明确、无超大上帝组件。
- 接口调用全部通过统一请求层。
- 路由、权限、缓存、异常提示行为统一。
- 关键页面具备可用性与基本测试覆盖。

### 8.2 后端验收标准

- 所有接口具备统一响应结构、参数校验、异常处理。
- 密码安全、鉴权完整、日志规范。
- Controller 不再承担复杂业务。
- 核心业务具备测试覆盖。

### 8.3 数据库验收标准

- 命名、索引、约束、审计字段规范统一。
- 核心业务表结构语义清晰、可扩展。
- 数据迁移方案明确且可执行。

### 8.4 工程化验收标准

- 项目可一键启动或按文档顺利启动。
- 具备规范文档、接口文档、数据库文档、部署说明。
- 可支持毕业设计展示与后续继续开发。
